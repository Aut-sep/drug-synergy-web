## 1. 本文档用途

这份文档用于持续记录本毕业设计在实际推进过程中的思路、问题、决策、已完成工作和阶段性结果，方便后续撰写毕业论文中的“研究背景”“方法设计”“实验方案”“问题分析”和“工作过程”部分。

这份记录不是单纯的开发日志，而是按毕业论文可用的表达方式整理，重点体现以下几点：

1. 先阅读原始论文，理解四个模型各自的方法。
2. 再发现四个模型之间不能直接公平比较的问题。
3. 然后设计统一的数据处理与对比方案。
4. 最后把方案落实为可重复使用的脚本和文档。

## 2. 当前课题目标

本课题的核心目标不是简单复现某一篇论文，而是：

**在尽量保留四个原始模型输入接口和原始代码风格的前提下，构建一套统一的数据适配与统一比较流程，用于评估四个药物协同预测模型在同一数据集、同一划分协议、同一评价指标下的表现。**

这个目标比“单独复现一个模型”更适合作为本科毕业设计，因为它既有论文阅读和方法分析工作，也有数据工程、脚本实现、实验设计和结果比较工作。

## 4. 研究推进过程

### 第一阶段：阅读四个模型原始论文

在这一阶段，主要工作是阅读四篇模型论文，梳理每个模型的：

1. 任务定义
2. 输入特征
3. 标签构造方式
4. 数据集来源
5. 训练与测试划分方式
6. 评价指标

这样做的原因是，只有先把四个模型“原来是怎么做的”搞清楚，后面才能判断它们是否适合放在同一个实验表中进行比较。

### 第二阶段：发现四个模型不能直接公平比较

阅读后发现，虽然这四个模型都属于药物组合协同预测方向，但它们原论文的实验条件并不一致，主要体现在以下几个方面：

1. 标签阈值不同  
   其中 `DualSyn`、`MFSynDCP`、`MVCASyn` 更接近使用 `synergy > 10` 作为正样本、`synergy < 0` 作为负样本；而 `MTLSynergy` 使用的正样本阈值更高。

2. 数据样本集合不同  
   各论文虽然大多基于 O'Neil 数据来源，但经过各自筛选、清洗和映射后，最终使用的样本数量、药物数量和细胞系数量并不完全相同。

3. 五折划分方式不同  
   有的论文是随机样本五折，有的是按药物组合分组五折，有的是先切出测试集再在训练集内部做五折。

4. 输入特征不同  
   有的模型只使用药物图和细胞表达特征，有的额外使用 copy number，有的还使用单药敏感性信息。

5. 任务定义不同  
   前三个模型更接近协同二分类任务，而 `MTLSynergy` 是协同预测和单药敏感性联合学习的多任务模型。

这说明一个关键问题：

**不能直接拿四篇论文中的原始结果表格横向比较。**

因为如果数据、标签、划分协议都不同，那么最终性能差异不一定来自模型结构本身，也可能来自实验设置差异。

### 第三阶段：明确毕业设计应该解决的问题

在确定四个模型不能直接公平比较后，本课题的重点从“直接做交集数据集”转变为：

**建立一套统一数据输入规范，再将同一份统一数据分别转换成四个模型可读取的格式。**

这样做的优点是：

1. 不依赖一次性的人工交集处理。
2. 后续如果更换数据集，不需要重写四套零散脚本。
3. 可以把“统一数据输入”和“模型专用格式导出”分离开来。
4. 更方便在论文中说明方法流程，逻辑更清楚。

### 第四阶段：新建独立目录，避免和旧处理流程混在一起

为了避免新旧代码混杂，已经单独建立了新的工作目录：

- `D:\codex\bishe_base\benchmark_factory`

这一目录的作用是：

1. 作为统一 benchmark 数据处理工厂。
2. 专门存放新的输入规范、数据适配脚本、说明文档和导出结果。
3. 不直接修改四个模型原始目录中的旧数据处理逻辑。

这样做的原因是，后续无论是调试、写论文还是给老师展示，都会更清晰，也更容易说明“这是为了统一比较而新增的一层适配系统”。

### 第五阶段：形成统一 benchmark 的核心方案

在综合四篇论文方法之后，目前已经形成如下统一思路：

1. 使用统一的 canonical input bundle 作为输入。
2. 输入 bundle 中保存样本表、药物表以及不同模型需要的细胞特征表。
3. 通过一套通用脚本读取输入 bundle。
4. 再分别导出为 `DualSyn`、`MFSynDCP`、`MVCASyn`、`MTLSynergy` 四个模型可直接读取的目录结构和文件格式。
5. 统一五折划分协议。
6. 统一主实验标签规则。
7. 统一主实验评价指标。

这意味着后续只要准备一份满足固定格式的数据集，就可以自动生成适配四个模型的数据版本，而不需要每次手工重做。

6. `论文方法说明_中文.md`  
   用论文写作语言解释为什么不能直接比较原论文结果，以及为什么要构建统一 benchmark。

7. `四模型原方法_比较问题_解决方案表格.md`  
   以表格形式整理四个模型原方法、比较困难和解决方式，适合直接支撑论文中的“问题发现与方案设计”部分。

### 5.2 已完成的代码能力

目前脚本已经具备以下能力：

1. 读取统一输入数据包。
2. 检查样本中的药物和细胞系是否能在对应特征文件中找到。
3. 生成统一的样本分配表。
4. 根据同一套样本生成四个模型各自可读取的导出数据目录。
5. 记录导出协议配置，便于后续实验复现。

### 5.3 已支持的折叠划分方式

当前脚本支持以下三种五折划分策略：

1. `pair_group`
2. `pair_cell_group`
3. `sample_group`

其中当前推荐作为**主实验默认方案**的是：

- `pair_group`

原因是它按“无序药物组合”分组，能够尽量避免同一药物组合同时出现在训练集和测试集，从而降低信息泄漏风险，更适合用于统一、公平的模型比较。

### 5.4 已完成的初步验证

当前脚本已经完成过基础可运行性验证，包括：

1. 核心 Python 脚本语法检查通过。
2. 基于示例数据包完成过导出测试。
3. 导出结果中已经能够记录当前使用的划分协议和实验配置。

这说明目前新的 benchmark 数据处理框架已经不是停留在想法层面，而是已经具备了实际可运行的基础。

## 6. 目前已经形成的主要结论

### 结论一：四个模型不能直接拿原论文结果做公平比较

这是目前最重要的研究判断之一。原因不是模型名字不同，而是：

1. 数据集范围不同
2. 标签规则不同
3. 五折协议不同
4. 输入信息量不同
5. 任务目标不同

因此，原论文结果只能用于了解模型设计思路，不能直接当作同一表格中的公平对比结果。

### 结论二：统一 benchmark 比直接做一次性交集更合理

最开始的思路更接近“做一个四模型交集数据集”，但在推进中发现，这种方法有两个问题：

1. 一次性交集过程不够透明，后续难以解释和复用。
2. 如果以后想换数据集，又要重新人工处理一次。

因此，更合理的做法是：

**先定义统一输入格式，再通过脚本自动导出四个模型对应格式。**

这样不仅更规范，也更符合毕业论文对“方法设计”和“可复现性”的要求。

### 结论三：本科毕业设计不必盲目追求很多来源数据集

结合参考本科论文的工作量水平，目前更适合本课题的路线是：

1. 以一个主数据集作为核心实验对象。
2. 在这个主数据集上完成四模型统一比较。
3. 通过补充实验、不同协议对比、不同阈值讨论来体现工作量。

这比强行再引入多个新数据集更稳妥，也更容易在有限时间内做扎实。

### 结论四：主数据集优先推荐 O'Neil

当前最适合作为统一主数据集的是：

- `O'Neil`

原因如下：

1. 四个模型都和 O'Neil 数据有直接或间接关系，论文背景更容易对齐。
2. 以 O'Neil 为主数据集，更容易解释“为什么要统一比较”。
3. 对本科毕业设计来说，一个主数据集配合多种实验维度已经足够构成完整工作量。

## 7. 当前推荐的毕业设计实验方案

结合目前工作进度和本科毕业设计的合理范围，当前推荐的实验结构如下。

### 7.1 主实验

在统一的 O'Neil 数据集上，将四个模型放到同一协议下进行比较：

1. 统一样本来源
2. 统一标签规则
3. 统一五折协议
4. 统一评价指标

这一部分将作为毕业论文最核心的实验内容。

### 7.2 补充实验

为了体现工作量和分析深度，可以在不明显增加项目风险的前提下增加以下补充内容：

1. 比较 `pair_group` 和 `sample_group` 两种五折方式下的结果差异。
2. 说明原论文设定与统一 benchmark 设定之间的差异。
3. 对 `MTLSynergy` 的多任务特点进行单独讨论，说明为什么主表中只比较其协同预测部分。

### 7.3 不建议当前阶段优先做的内容

现阶段不建议把主要精力放在以下方向：

1. 同时强行做很多新的来源数据集。
2. 过早追求把所有论文所有扩展实验全部原样重跑。
3. 在没有统一输入规范前继续做零散的人工交集处理。

原因是这些内容会明显提高工作量和不确定性，但对本科毕业论文的增益不一定成比例。

## 10. 后续建议执行顺序

为了让接下来的工作更稳，建议按下面顺序推进：

1. 先最终确定主数据集使用 O'Neil。
2. 把 O'Neil 整理成统一输入 bundle 所要求的固定格式。
3. 使用 `benchmark_factory` 生成四个模型各自的数据目录。
4. 统一四个模型的训练入口和五折执行方式。
5. 先完成主实验。
6. 再补充少量扩展实验和讨论内容。
7. 最后统一汇总指标，形成论文中的结果表和分析文字。

这个顺序的优点是：

1. 先把主线做通。
2. 先保证论文一定能交付。
3. 再在有余力时补工作量，而不是一开始就把任务铺得太大。
