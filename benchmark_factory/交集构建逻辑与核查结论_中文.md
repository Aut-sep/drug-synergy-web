# 交集构建逻辑与核查结论

更新时间：2026-04-13

## 1. 文档目的

这份文档用于说明：

1. 四模型统一交集数据集是如何构建出来的。
2. 在交集构建过程中具体检查了哪些潜在问题。
3. 为什么当前这版严格交集数据集可以作为后续毕业论文主实验的数据基础。

这份文档对应的最终推荐数据集是：

- `benchmark_factory/input_bundles/oneil_intersection_strict_gt10_lt0`

---

## 2. 最终采用的数据集定义

当前最终推荐的数据集不是完整 `O'Neil`，而是：

**完整 O'Neil 数据中，四个模型在当前本地原始输入条件下都能读取的共同交集，并进一步采用严格标签规则筛选后的样本集。**

严格标签规则为：

1. `synergy_score > 10` 记为正样本 `1`
2. `synergy_score < 0` 记为负样本 `0`
3. `0 <= synergy_score <= 10` 的样本全部删除

最终数据集规模为：

1. 药物数：`38`
2. 细胞系数：`29`
3. 样本数：`12283`
4. 正样本数：`5764`
5. 负样本数：`6519`

---

## 3. 交集构建所用源数据

交集构建的起点不是之前人工做过的旧交集文件，而是以下本地原始源数据：

1. `MTLSynergy/data/oneil_summary_idx_original_backup.csv`
2. `MTLSynergy/data/drugs.csv`
3. `MTLSynergy/data/cell_lines.csv`

这样做的原因是：

1. `oneil_summary_idx_original_backup.csv` 含有 `22737` 条样本，规模与论文和项目说明更一致。
2. 当前 `MTLSynergy/data/oneil_summary_idx.csv` 只有 `7144` 条，更像后处理子集，不适合继续当作完整 O'Neil 源表。

因此，交集构建首先基于 `oneil_summary_idx_original_backup.csv` 还原出完整的：

1. O'Neil 药物集合
2. O'Neil 细胞系集合
3. O'Neil 样本集合

---

## 4. 交集构建的核心逻辑

### 4.1 第一步：确定完整 O'Neil 中实际出现的药物和细胞系

根据 `oneil_summary_idx_original_backup.csv` 中的：

1. `drug_row_idx`
2. `drug_col_idx`
3. `cell_line_idx`

再结合：

1. `drugs.csv`
2. `cell_lines.csv`

还原出完整 O'Neil 中实际使用的药物和细胞系。

这里得到的完整 O'Neil 规模是：

1. `38` 个药物
2. `39` 个细胞系

### 4.2 第二步：统一药物名和细胞名

为了避免把“名字不同但实际是同一对象”的情况误判成缺失，在做交集前先做标准化映射。

当前实际使用的映射有：

```text
Carboplatinum -> CARBOPLATIN
NIHOVCAR3 -> OVCAR3
```

解释如下：

1. 完整 O'Neil 源表中的铂类药物使用的是 `Carboplatinum`
2. 另外三个模型的药物文件普遍使用 `CARBOPLATIN`
3. `MVCASyn` 的细胞特征文件里使用 `NIHOVCAR3`
4. 统一 O'Neil 源表里对应名字是 `OVCAR3`

因此，如果不先做名称统一，会错误缩小交集。

### 4.3 第三步：求四模型共同可用药物集合

药物交集的判断依据是：

1. 该药物必须出现在完整 O'Neil 样本中
2. 必须能在 `DualSyn` 的 `smiles.csv` 中找到
3. 必须能在 `MFSynDCP` 的 `smiles.csv` 中找到
4. 必须能在 `MVCASyn` 的 `oneil_drug_two_smiles.csv` 中找到
5. 必须能在 `MTLSynergy` 的 O'Neil 索引系统中找到对应药物行

核查结果表明：

1. 药物层面没有真实缺失
2. 真正需要处理的是名称不一致，而不是药物不存在

因此最终共同可用药物数为：

- `38`

### 4.4 第四步：求四模型共同可用细胞系集合

细胞交集的判断依据是：

1. 该细胞系必须出现在完整 O'Neil 样本中
2. 必须能在 `DualSyn` 的 `cell_features_954.csv` 中找到
3. 必须能在 `MFSynDCP` 的 `cell_features.csv` 中找到
4. 必须同时出现在 `MVCASyn` 的 `exp.csv` 和 `cn.csv` 中
5. 必须能在 `MTLSynergy` 的索引系统中找到

注意：

1. `DualSyn` 的细胞名带组织后缀，例如 `A2058_SKIN`
2. 因此在交集判断时需要先截取下划线前的主细胞系名

按这个规则求交后，最终共同可用细胞系数为：

- `29`

### 4.5 第五步：筛选共同样本

样本保留规则是：

1. 一条样本中的 `drug_a` 必须在共同药物集合中
2. `drug_b` 也必须在共同药物集合中
3. `cell_line` 必须在共同细胞系集合中

满足以上三点的样本才会保留。

按这个规则，从完整 `22737` 条 O'Neil 样本中保留下来的原始交集样本数为：

- `16907`

### 4.6 第六步：应用严格标签规则

在 `16907` 条原始交集样本上，进一步应用论文主实验的严格标签规则：

1. `synergy > 10` 保留并标记为 `1`
2. `synergy < 0` 保留并标记为 `0`
3. `0~10` 区间样本全部删除

删除后最终得到：

- `12283` 条样本

这就是当前最终使用的主实验数据集。

## 5. 本次核查中重点检查的问题

为了确认这套交集不要轻易再改，本次除了做普通集合运算，还额外检查了几个最容易出问题的点。

### 5.1 是否存在样本重复

对最终严格版样本表检查了：

1. 有序三元组 `(drug_a, drug_b, cell_line)` 是否重复
2. 无序药物对三元组 `({drug_a, drug_b}, cell_line)` 是否重复

核查结果：

1. 有序三元组重复数：`0`
2. 无序三元组重复数：`0`

这说明最终保留样本中，没有同一药物组合在同一细胞系上的重复记录。

### 5.2 是否完整保留了 O'Neil 的药物对结构

在最终严格版数据集中，统计到的唯一无序药物对数量为：

- `583`

这与 O'Neil 常见描述中的 `583` 个药物组合是一致的。

这说明：

1. 虽然删除了一部分中间区间样本
2. 但整体药物对空间没有被异常破坏

### 5.3 药物别名是否会误取错 MTLSynergy 行

这一项是本次核查里比较重要的异常点。

在 `MTLSynergy/data/drugs.csv` 中，标准化后会出现一个特殊情况：

```text
CARBOPLATIN <- id 5, name = Carboplatinum, from_oneil = No
CARBOPLATIN <- id 280, name = CARBOPLATIN, from_oneil = Yes
```

表面上看，似乎应该优先使用 `id=280`。

但继续检查 `oneil_summary_idx_original_backup.csv` 后发现：

1. 完整 O'Neil 样本实际引用的是 `id=5`
2. 并没有引用 `id=280`

这说明：

1. 本地 `MTLSynergy` 原始数据中，`from_oneil` 标记和 `summary` 实际引用关系并不完全一致
2. 真正可靠的判断标准不是 `from_oneil`
3. 而是“该药物行是否真的被 O'Neil 的 summary 索引表使用”

因此，当前交集构建脚本已经改为：

**若标准化后出现多个同名候选，则优先选择在 `summary` 中真实被引用的那一行。**

这样做比单纯看名字或看 `from_oneil` 标记更稳。

### 5.4 DualSyn 与 MFSynDCP 的 954 维细胞矩阵是否一致

这项检查也很重要。

本来以为 `DualSyn` 和 `MFSynDCP` 的 954 维细胞特征表应当完全一致，但实际核对发现：

1. 29 个共同细胞系里，有 28 个的数值一致
2. 只有 `NCIH23` 这一行数值明显不一致

进一步检查发现：

1. `DualSyn` 中 `NCIH23` 的数值正常对应 `NCIH23_LUNG`
2. `MFSynDCP` 中标成 `NCIH23` 的这一行，数值却与 `DualSyn` 里的 `NCIH2342` 完全一致

这说明 `MFSynDCP` 本地 `cell_features.csv` 中的 `NCIH23` 行很可能存在历史标注错误或行错配问题。

因此，当前统一交集 bundle 在构建共享 954 维细胞矩阵时，采用的是：

- `DualSyn/data/cell_features_954.csv`

而不是 `MFSynDCP/data/cell_features.csv`。

这样做的原因是：

1. 可以避免把明显可疑的 `NCIH23` 行继续传播到统一 benchmark 中
2. 又不需要因为这 1 个可疑行把整个 `NCIH23` 细胞系从交集中彻底删掉

需要说明的是：

1. 这不会改变交集“哪些样本被保留”
2. 它只影响统一 bundle 中 954 维细胞特征从哪份原始表导出

---

## 6. 为什么当前交集逻辑可以定版

经过上述核查，当前这套交集逻辑可以认为是稳定可用的，理由如下：

1. 源数据不是基于旧人工交集，而是基于完整 O'Neil 本地备份重新构建
2. 药物和细胞名在取交集前已经做了必要的别名统一
3. 交集条件直接对应四个模型原始代码的真实输入要求
4. 最终样本中不存在重复三元组
5. 药物对总数仍保持 `583`
6. 对本地原始数据中的特殊异常做了额外修正
7. 生成后的标准 bundle 已经能够成功导出为四个模型格式

因此，当前建议是：

**后续主实验应固定使用 `oneil_intersection_strict_gt10_lt0` 这套数据集，不再随意改动交集规则。**

## 7. 最终可直接写进论文的方法性表述

下面这段话可以直接改写后放进论文正文：

> 本文首先以本地保存的完整 O'Neil 组合药物筛选样本表为基础，结合药物与细胞系索引表还原出完整的药物集合、细胞系集合及样本集合。随后，依据 DualSyn、MFSynDCP、MVCASyn 和 MTLSynergy 四个模型原始代码的真实输入要求，对药物 SMILES、细胞表达特征和拷贝数特征的可用性进行逐项核对，并在名称层面对 Carboplatinum/CARBOPLATIN 和 NIHOVCAR3/OVCAR3 等别名进行统一。只有当一条样本中的两个药物和对应细胞系均能被四个模型的输入系统同时支持时，该样本才被保留。通过该规则，完整 O'Neil 数据被约束为包含 38 个药物、29 个细胞系的共同交集样本集。在此基础上，进一步采用严格标签策略，仅保留协同分数大于 10 的正样本和小于 0 的负样本，删除中间区间样本，最终得到 12283 条用于四模型统一比较的标准数据集。为避免局部原始文件中存在的可疑特征行影响统一比较，本文还对交集构建过程中的药物别名冲突和单个细胞系特征错配问题进行了专项核查，并据此固定了统一 benchmark 的导出规则。
