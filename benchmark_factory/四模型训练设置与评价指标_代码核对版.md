## 3. DualSyn

### 3.1 训练脚本

主实验脚本：

- `DualSyn/DualSyn/train_transductive.py`

### 3.2 代码中可直接确认的训练设置

从 `train_transductive.py` 可直接读出：

1. 训练轮数：`NUM_EPOCHS = 400`
2. 学习率：`LR = 0.0005`
3. 训练 batch size：`TRAIN_BATCH_SIZE = 1024`
4. 测试 batch size：`TEST_BATCH_SIZE = 1024`
5. 损失函数：`nn.BCELoss()`
6. 优化器：`torch.optim.Adam(model.parameters(), lr=LR)`
7. 设备：
   - 若 `torch.cuda.is_available()`，则使用 `cuda:0`
   - 否则使用 `cpu`
8. 数据读取：
   - `data/cell_features_954.csv`
   - `data/smiles.csv`
   - `data/fold{i}/train.csv`
   - `data/fold{i}/test.csv`
9. 训练折数：代码中 `for i in range(5)`，即五折

### 3.3 代码中实际计算的指标

`DualSyn` 在测试时实际计算了：

1. `AUC`：`roc_auc_score`
2. `PR_AUC`：`precision_recall_curve + metrics.auc`
3. `ACC`：`accuracy_score`
4. `BACC`：`balanced_accuracy_score`
5. `PREC`：`precision_score`
6. `TPR`：由混淆矩阵计算 `tp / (tp + fn)`
7. `KAPPA`：`cohen_kappa_score`
8. `RECALL`：`recall_score`

同时，它还会打印训练集上的：

1. `train_AUC`
2. `train_PR_AUC`
3. `train_ACC`

### 3.4 代码中的模型保存逻辑

这一点很重要。

`DualSyn` 代码里虽然维护了 `best_auc`，但：

1. 它只是把更优的 AUC 写入结果文件
2. 最后真正 `torch.save` 的是每折训练结束后的最终模型
3. 不是“最优 AUC 对应的模型”

因此，按代码实现看：

**DualSyn 主实验是“记录最佳 AUC，但最终保存最后一轮模型权重”。**

### 3.5 对论文写作很重要的说明

1. `DualSyn` 主实验脚本没有计算 `F1`
2. 也没有在代码里做显式 early stopping
3. 主实验的结果记录核心是每个 epoch 的测试折 AUC 等指标

## 4. MFSynDCP

### 4.1 训练脚本

主实验流程：

1. `MFSynDCP/MFSynDCP/creat_data.py`
2. `MFSynDCP/MFSynDCP/training.py`

其中：

1. `creat_data.py` 负责把五折 `train.csv/test.csv` 转成 `.pt` 缓存
2. `training.py` 负责正式训练和测试

### 4.2 代码中可直接确认的训练设置

从 `training.py` 可直接读出：

1. 训练轮数：`NUM_EPOCHS = 500`
2. 学习率：`LR = 0.0001`
3. 训练 batch size：`TRAIN_BATCH_SIZE = 128`
4. 脚本里定义了 `TEST_BATCH_SIZE = 256`
5. 但实际测试 DataLoader 用的仍然是 `TRAIN_BATCH_SIZE`
6. 损失函数：`nn.CrossEntropyLoss()`
7. 优化器：`torch.optim.Adam(model.parameters(), lr=LR)`
8. 设备：
   - `os.environ["CUDA_VISIBLE_DEVICES"] = "1"`
   - 若有 CUDA，则使用 `cuda`
   - 否则使用 `cpu`
9. 训练折数：代码中 `for i in range(5)`，即五折

### 4.3 代码中实际计算的指标

`MFSynDCP` 在测试时实际计算了：

1. `AUC`
2. `PR_AUC`
3. `ACC`
4. `BACC`
5. `PREC`
6. `TPR`
7. `KAPPA`
8. `RECALL`

这里和 `DualSyn` 类似：

1. 使用 `roc_auc_score`
2. 使用 `precision_recall_curve + metrics.auc`
3. 使用 `balanced_accuracy_score`
4. 使用 `cohen_kappa_score`

### 4.4 代码中的模型保存逻辑

`MFSynDCP` 与 `DualSyn` 不同。

它的代码逻辑是：

1. 每个 epoch 在测试折上计算 `AUC`
2. 如果当前 `AUC` 超过 `best_auc`
3. 就立刻保存当前模型参数

因此：

**MFSynDCP 是按“测试折 AUC 最优”保存模型。**

### 4.5 对论文写作很重要的说明

1. `TEST_BATCH_SIZE` 虽然被定义为 `256`
2. 但测试 DataLoader 实际仍用了 `TRAIN_BATCH_SIZE = 128`
3. 因此论文中如果写 batch size，应该谨慎表述：
   - 代码定义的训练 batch size 为 128
   - 实际训练和测试 DataLoader 都使用了 128

另外：

1. `MFSynDCP` 主实验代码没有显式计算 `F1`
2. 也没有早停机制

## 5. MVCASyn

### 5.1 训练脚本

主实验脚本：

- `MVCASyn/cv_train.py`

### 5.2 代码中可直接确认的训练设置

`MVCASyn` 的训练参数是通过命令行参数定义的，默认值在代码中可直接看到：

1. 原子输入特征数：`n_atom_feats = 55`
2. 原子隐藏维度：`n_atom_hid = 128`
3. 交互矩阵维度：`kge_dim = 128`
4. pooling ratio：`0.6`
5. 卷积通道数：
   - `conv_channel1 = 16`
   - `conv_channel2 = 16`
6. 学习率：`lr = 0.001`
7. 训练轮数：`n_epochs = 125`
8. batch size：`512`
9. weight decay：`5e-4`
10. 设备：
    - 若有 CUDA 且 `use_cuda=True`，则使用 `cuda:0`
    - 否则使用 `cpu`
11. 学习率调度器：
    - `LambdaLR(optimizer, lambda epoch: 0.96 ** epoch)`
12. 优化器：
    - `optim.Adam(model.parameters(), lr=lr, weight_decay=weight_decay)`
13. 损失函数：
    - `custom_loss.SigmoidLoss()`

### 5.3 自定义损失函数

在 `MVCASyn/custom_loss.py` 中，`SigmoidLoss` 的定义为：

1. 正样本损失：`-logsigmoid(p_score).mean()`
2. 负样本损失：`-logsigmoid(-n_score).mean()`
3. 总损失：两者平均

因此：

**MVCASyn 使用的是项目自定义的 SigmoidLoss，而不是标准 BCE 或 CE。**

### 5.4 代码中实际计算的指标

`MVCASyn` 代码中实际计算并保存了以下测试指标：

1. `test_acc`
2. `test_auc_roc`
3. `test_int_ap`
4. `test_ap`
5. `test_precision`
6. `test_kappa`
7. `test_f1`
8. `test_recall`
9. `test_bacc`
10. `test_prec`
11. `test_tnr`

这些指标分别对应：

1. `acc`：`accuracy_score`
2. `auroc`：`roc_auc_score`
3. `int_ap`：`precision_recall_curve` 曲线积分，也就是 PR-AUC
4. `ap`：`average_precision_score`
5. `precision`：`precision_score`
6. `kappa`：`cohen_kappa_score`
7. `f1_score`：`f1_score`
8. `recall`：`recall_score`
9. `tnr`：真负率
10. `prec`：`sum(target) / len(target)`，即样本中正类比例

### 5.5 代码中 `BACC` 的特殊实现

这一点非常重要。

`MVCASyn` 的代码里：

```python
bacc = (recall + (1 - precision)) / 2
```

也就是说，它的 `BACC` 并不是 `sklearn.metrics.balanced_accuracy_score` 的标准实现。

因此，论文里如果要写这一点，必须如实写成：

**MVCASyn 代码中记录了名为 `BACC` 的指标，但其计算公式是 `(recall + (1 - precision)) / 2`，并非 sklearn 的标准 balanced accuracy 实现。**

### 5.6 代码中的模型保存逻辑

`MVCASyn` 的模型保存逻辑也比较特别：

1. 训练过程中每个 epoch 都计算训练集指标
2. 如果当前 `train_acc` 超过历史最大值
3. 就保存模型

然后测试阶段是：

1. 训练结束后加载这个保存的模型
2. 在测试集上评估

因此：

**MVCASyn 是按训练集 ACC 最优来保存模型，而不是按验证集或测试集指标。**

这个实现特点在论文里最好单独说明。

## 6. MTLSynergy

### 6.1 训练脚本

主实验对应两个脚本：

1. `MTLSynergy/AEtrain.py`
2. `MTLSynergy/MTLSynergytrain.py`

其中：

1. `AEtrain.py` 先训练药物和细胞系自编码器
2. `MTLSynergytrain.py` 再使用预训练编码器进行多任务训练

### 6.2 AEtrain.py 中可直接确认的训练设置

#### 药物自编码器

1. 输入数据：`data/drug_features.csv`
2. batch size：`32`
3. 学习率：`0.0001`
4. 最大 epoch：`3000`
5. patience：`100`
6. 损失函数：`MSELoss(reduction='mean')`
7. 优化器：`Adam`
8. 输出维度候选：
   - `[32, 64, 128, 256, 512]`

#### 细胞系自编码器

1. 输入数据：`data/cell_line_features.csv`
2. batch size：`32`
3. 学习率：`0.0001`
4. 最大 epoch：`1500`
5. patience：`100`
6. 损失函数：`MSELoss(reduction='mean')`
7. 优化器：`Adam`
8. 输出维度候选：
   - `[128, 256, 512, 1024, 2048]`

#### 设备

代码中直接写为：

- `device = torch.device('cuda')`

因此，按原始代码：

**AEtrain.py 默认要求 CUDA 环境。**

### 6.3 MTLSynergytrain.py 中可直接确认的训练设置

从代码中可直接读出：

1. epoch：`500`
2. patience：`100`
3. batch size：`256`
4. 损失函数：
   - 回归部分：`MSELoss`
   - 分类部分：`CategoricalCrossEntropyLoss`
5. 优化器：`Adam`
6. 设备：
   - `device = torch.device('cuda')`
7. 预训练编码器维度来自：
   - `DrugAE_OutputDim = 128`
   - `CellAE_OutputDim = 256`

### 6.4 超参数搜索

`MTLSynergytrain.py` 中显式定义了两个候选超参数组：

1. 学习率 `0.0001`
   - hidden layers: `[8192, 4096, 4096, 2048]`
2. 学习率 `0.0001`
   - hidden layers: `[4096, 2048, 2048, 1024]`

代码流程是：

1. 对每个外层测试折 `fold_test`
2. 在剩余 4 折上继续做内层折搜索
3. 按验证 loss 选最优超参数
4. 再在外层测试折上评估

因此：

**MTLSynergy 不是简单固定参数五折，而是带有内层超参数选择的外层五折流程。**

### 6.5 Dataset.py 中非常关键的标签设置

在 `MTLSynergy/Dataset.py` 中：

1. `MTLSynergyDataset(..., syn_threshold=30, ri_threshold=50)`

也就是说，默认情况下：

1. 协同标签阈值使用 `30`
2. 药物敏感性标签阈值使用 `50`

这是从代码里直接能确认的设置。

因此，论文写作时必须注意：

**MTLSynergy 原始代码默认的协同二分类阈值不是 `10`，而是 `30`。**

### 6.6 代码中实际计算的指标

`MTLSynergy` 的测试结果分成两部分：

1. `Synergy`
2. `Sensitivity`

对两部分都计算了：

1. `MSE`
2. `RMSE`
3. `Pearsonr`
4. `CCE`
5. `ROC AUC`
6. `PR AUC`
7. `ACC`
8. `PREC`
9. `Kappa`

代码中没有为主测试函数显式计算：

1. `Recall`
2. `F1`
3. `BACC`

因此，若论文中统一比较四模型分类结果：

**MTLSynergy 原始代码直接给出的分类指标主要是 `ROC AUC`、`PR AUC`、`ACC`、`PREC`、`Kappa`。**

### 6.7 代码中的模型保存逻辑

`MTLSynergy` 使用 `EarlyStopping`，按验证总损失保存模型。

其 `EarlyStopping` 逻辑为：

1. 若当前验证 loss 更小，则保存模型参数
2. 若连续 `patience` 次没有改善，则 early stop

因此：

**MTLSynergy 是按验证 loss 最优来保存模型，而不是按 AUC。**

## 9. 写进论文时最应该强调的几点

如果你要体现“认真读过代码”，下面几点最值得写：

1. `DualSyn` 和 `MFSynDCP` 的主实验指标集合很接近，都包含 `AUC`、`PR_AUC`、`ACC`、`BACC`、`PREC`、`Kappa` 等。
2. `MVCASyn` 使用的是自定义 `SigmoidLoss`，并非标准 BCE/CE。

6. `MTLSynergy` 先训练自编码器，再训练多任务模型。
7. `MTLSynergy` 原始代码默认协同分类阈值是 `30`，不是 `10`。
8. `MTLSynergy` 的模型选择标准是验证损失，而不是 AUC。

这些都是代码里能直接确认、而且很能体现“确实认真看过仓库实现”的信息。

## 10. 可直接写进论文的方法性表述

下面这段话可以直接改写后放进论文：

> 为避免仅依据论文文字对模型训练细节进行概括，本文进一步对四个模型的公开代码进行了逐项核对。核查结果表明，DualSyn 主实验脚本采用 Adam 优化器和 BCELoss，按五折 transductive 方式训练 400 个 epoch，并在每轮测试折上计算 AUC、PR-AUC、ACC、BACC、Precision、Kappa 和 Recall 等指标；MFSynDCP 的训练脚本采用 Adam 和 CrossEntropyLoss，训练轮数为 500，主测试指标与 DualSyn 基本一致，但其代码实际按测试折 AUC 保存最优模型；MVCASyn 采用自定义 SigmoidLoss、Adam 优化器和学习率衰减调度器，默认训练 125 个 epoch，并输出 AUC、PR-AUC、Average Precision、F1、Kappa 等指标，但其代码中记录的 BACC 并非 sklearn 标准 balanced accuracy 实现；MTLSynergy 则首先通过自编码器预训练药物和细胞系表征，再在多任务框架下联合优化协同预测和敏感性预测，其测试阶段同时输出回归指标与分类指标，并使用验证损失结合 early stopping 选择模型。上述差异说明，即使四个模型都用于药物组合协同预测，其训练控制方式、模型选择标准与指标实现细节仍存在显著差别，因此在统一 benchmark 比较中必须显式固定数据、划分协议和评价口径。
