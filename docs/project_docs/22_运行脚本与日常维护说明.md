# 运行脚本与日常维护说明

## 1. 脚本清单

项目根目录 `scripts/` 下当前主要脚本如下：

| 脚本 | 主要用途 |
| --- | --- |
| `start_web_system.ps1` | 本地启动前端、后端，并按需启动推理栈与训练服务 |
| `start_web_system_docker.ps1` | 容器化启动前端与后端 |
| `check_web_system.ps1` | 检查前端、后端、推理网关和训练服务的可达性 |

## 2. `start_web_system.ps1`

### 2.1 作用

该脚本面向本地开发和答辩演示环境，负责：

1. 检查 Docker Desktop 是否就绪。
2. 在需要时启动推理运行时栈。
3. 启动本地后端。
4. 启动本地前端。
5. 按需在 WSL 中拉起训练服务。

### 2.2 常用命令

默认启动：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system.ps1
```

同时启动训练服务：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system.ps1 -StartTrainingService
```

跳过 Docker 检查与推理栈启动：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system.ps1 -SkipDocker
```

### 2.3 使用注意

1. 脚本启动推理栈时使用 `docker compose ... up -d --no-build`。
2. 因此首次运行前，应先手动构建 `web_system_runtime/docker-compose.yml` 对应镜像。
3. 若 `5173`、`9000`、`8000`、`8011` 已有服务占用，脚本会优先判断端口是否已监听。

## 3. `start_web_system_docker.ps1`

### 3.1 作用

该脚本面向容器化 Web 层启动场景，负责：

1. 检查 Docker Daemon。
2. 使用 `docker-compose.web.yml` 启动 `web-backend` 和 `web-frontend`。

### 3.2 常用命令

默认启动并构建：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system_docker.ps1
```

跳过构建：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system_docker.ps1 -NoBuild
```

### 3.3 边界说明

该脚本只负责 Web 前后端容器，不负责：

1. 推理栈容器构建与长期维护。
2. WSL 训练服务启动。

如果推理栈尚未启动，需要额外执行：

```powershell
docker compose -f .\web_system_runtime\docker-compose.yml up -d --build
```

## 4. `check_web_system.ps1`

### 4.1 作用

该脚本用于快速巡检下列地址：

1. `http://127.0.0.1:5173`
2. `http://127.0.0.1:9000/health`
3. `http://127.0.0.1:9000/api/system/summary`
4. `http://127.0.0.1:8000/health`
5. `http://127.0.0.1:8011/health`

### 4.2 常用命令

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check_web_system.ps1
```

指定超时时间：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check_web_system.ps1 -TimeoutSeconds 10
```

### 4.3 结果判定

1. 前端、后端、系统概览和推理网关为必检项。
2. 训练服务为非必检项。
3. 任一必检项失败，脚本返回非零退出码，适合作为演示前检查或自动化脚本前置条件。

## 5. 日常维护建议

### 5.1 启动顺序建议

推荐顺序如下：

1. 构建并启动推理栈。
2. 启动 Web 前后端。
3. 如需训练演示，再启动 WSL 训练服务。
4. 最后执行健康检查脚本。

### 5.2 日志检查位置

| 路径 | 说明 |
| --- | --- |
| `web_system_backend/runtime_logs/backend.out.log` | 后端标准输出 |
| `web_system_backend/runtime_logs/backend.err.log` | 后端错误输出 |
| `web_system_frontend/runtime_logs/frontend.out.log` | 前端标准输出 |
| `web_system_frontend/runtime_logs/frontend.err.log` | 前端错误输出 |

### 5.3 运行目录维护

建议定期关注以下目录空间占用：

1. `web_system_runtime/user_bundles/web_system_uploads/`
2. `web_system_runtime/outputs/web_system_bridge/`
3. `web_system_runtime/trained_model_versions/`
4. `web_system_backend/outputs/downloads/`

### 5.4 数据库维护

`web_system_backend/data/web_system.db` 保存系统元数据，建议：

1. 在重要演示前备份该文件。
2. 在迁移机器或重装环境前先备份。
3. 定期核对任务记录与运行目录是否一致。

## 6. 常见问题排查

### 6.1 前端无法访问

优先检查：

1. `5173` 端口是否被占用。
2. `frontend.err.log` 是否存在构建错误。
3. 是否已执行 `npm install`。

### 6.2 后端无法访问

优先检查：

1. `9000` 端口是否被占用。
2. Python 依赖是否已安装。
3. `backend.err.log` 是否报出导入或配置错误。

### 6.3 推理任务无法提交

优先检查：

1. `8000/health` 是否可访问。
2. `web_system_runtime/docker-compose.yml` 对应容器是否启动。
3. 数据集是否通过样本校验并具备 `sample_file`。

### 6.4 训练任务无法提交

优先检查：

1. `8011/health` 是否可访问。
2. WSL 训练服务是否成功启动。
3. 数据集是否包含训练必需文件。

### 6.5 结果或产物无法下载

优先检查：

1. 后端接口返回的 `output_path_host` 是否存在。
2. 推理结果文件或训练产物目录是否已生成。
3. 后端下载目录是否可写。

## 7. 当前维护原则

1. 活动系统以原生 Web 工程为准。
2. 所有运行、检查和排障操作都围绕 `web_system_frontend/`、`web_system_backend/` 和 `web_system_runtime/` 展开。
