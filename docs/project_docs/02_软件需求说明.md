# 软件需求说明

## 1. 文档目的

本文档用于说明药物组合协同预测系统的功能需求、非功能需求、角色需求和约束条件，为系统设计、实现、测试和验收提供依据。

## 2. 系统目标

系统应支持四个药物协同预测模型在统一框架下完成数据组织、推理任务提交、训练任务管理、结果查看和版本追踪。

## 3. 用户角色

### 3.1 普通使用者

负责上传数据、提交推理任务、查看结果和浏览模型版本。

### 3.2 研究使用者

负责组织训练任务、调整训练参数、查看训练记录和导出实验结果。

### 3.3 系统维护者

负责检查服务状态、维护目录结构、核查任务执行情况和处理异常记录。

## 4. 功能需求

### 4.1 系统总览

1. 系统应展示数据集数量、任务数量、模型版本数量等概览信息。
2. 系统应展示推理服务组件、训练服务组件和存储目录的基本状态。
3. 系统应支持查看最近任务和关键运行指标。

### 4.2 数据集管理

1. 系统应支持上传标准化数据包。
2. 系统应支持对数据包进行基础合法性校验。
3. 系统应支持查看数据集元信息、文件组成和创建时间。
4. 系统应支持将数据集与后续任务建立关联。

### 4.3 推理任务管理

1. 系统应支持用户选择模型、数据集和运行参数。
2. 系统应支持创建推理任务并生成唯一任务编号。
3. 系统应支持查看任务状态，包括待执行、执行中、成功、失败、已取消。
4. 系统应支持查看推理输出文件、结果摘要和日志信息。
5. 系统应支持取消尚未结束的推理任务。

### 4.4 训练任务管理

1. 系统应支持用户选择模型、训练数据和训练参数。
2. 系统应支持创建训练任务并登记任务元信息。
3. 系统应支持查看训练任务状态、任务详情和训练产物摘要。
4. 系统应支持将训练结果登记到训练版本目录。
5. 系统应支持取消尚未结束的训练任务。

### 4.5 模型版本管理

1. 系统应支持按模型、时间和版本编号查看训练版本。
2. 系统应支持查看每个版本的参数摘要、指标摘要和文件清单。
3. 系统应支持将版本与来源任务建立可追溯关系。

### 4.6 结果与日志管理

1. 系统应支持按任务查看结果文件。
2. 系统应支持保存日志、指标和说明文件。
3. 系统应支持在系统工作目录中按规则组织输出。

## 5. 非功能需求

### 5.1 易用性

1. 页面结构应清晰，任务创建流程应尽量简洁。
2. 关键页面应提供说明文字，便于答辩演示和论文截图。

### 5.2 可维护性

1. 数据目录、任务目录和版本目录应保持稳定命名规则。
2. 模块职责应清晰，便于后续扩展模型或页面。

### 5.3 可追溯性

1. 每个任务应具有唯一标识。
2. 每个版本应能够追溯到来源任务、参数和结果文件。

### 5.4 可扩展性

1. 系统应便于加入新的模型执行组件。
2. 系统应便于扩展新的数据字段、页面或接口。

### 5.5 可靠性

1. 服务异常时应能够返回明确错误信息。
2. 任务失败时应保留日志与状态记录，便于后续排查。

## 6. 数据需求

1. 系统需要统一数据包描述药物、细胞系、标签和样本索引信息。
2. 系统需要保存数据集元信息、任务记录和版本记录。
3. 系统需要保存推理结果、训练结果和过程日志。

## 7. 接口需求

1. 系统应提供总览接口。
2. 系统应提供数据集上传与列表接口。
3. 系统应提供推理任务创建、列表、详情、取消接口。
4. 系统应提供训练任务创建、列表、详情、取消接口。
5. 系统应提供模型版本查询接口。

## 8. 运行约束

1. 当前设备可完成页面、接口、目录和轻量任务链路验证。
2. 完整训练执行受算力、依赖环境和运行时长影响。
3. 系统设计必须完整体现训练入口、训练参数、训练版本和训练产物组织方式。

## 9. 验收需求

1. 能够正常进入系统主要页面。
2. 能够查看系统总览信息。
3. 能够提交并查询推理任务。
4. 能够查看训练任务入口、任务记录和版本目录信息。
5. 能够通过文档明确说明系统架构、模块职责和现实限制。

## 10. 论文表述建议

在论文中可将需求分析总结为：

“系统需求的核心在于，将四个输入结构不同、执行路径不同的药物协同预测模型纳入统一的软件框架中，实现数据、任务、结果和版本的一体化管理。”
