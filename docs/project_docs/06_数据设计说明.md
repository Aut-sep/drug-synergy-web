# 数据设计说明

## 1. 设计目标

数据设计的目标是用统一的数据组织方式支撑四个模型的输入、推理、训练与结果管理，降低不同模型之间的数据结构差异对系统使用造成的影响。

## 2. 数据对象分类

系统中的核心数据对象包括：

1. 数据集对象。
2. 统一数据包对象。
3. 推理任务对象。
4. 训练任务对象。
5. 模型版本对象。
6. 推理结果对象。
7. 训练产物对象。

## 3. 统一数据包设计

### 3.1 设计目的

统一数据包用于抽象四模型的共同输入信息，为推理和训练提供一致的数据入口。

### 3.2 核心字段

1. 数据集标识。
2. 药物特征数据。
3. 细胞系特征数据。
4. 样本配对信息。
5. 协同标签信息。
6. 元信息说明文件。

### 3.3 设计原则

1. 公共字段优先统一。
2. 模型特定字段采用扩展区描述。
3. 文件命名和目录结构保持稳定。

## 4. 推理任务数据设计

推理任务记录建议包括以下字段：

| 字段 | 含义 |
| --- | --- |
| `task_id` | 推理任务编号 |
| `model_name` | 模型名称 |
| `dataset_id` | 数据集编号 |
| `status` | 任务状态 |
| `created_at` | 创建时间 |
| `finished_at` | 完成时间 |
| `output_dir` | 输出目录 |
| `result_summary` | 结果摘要 |
| `log_path` | 日志文件位置 |

## 5. 训练任务数据设计

训练任务记录建议包括以下字段：

| 字段 | 含义 |
| --- | --- |
| `task_id` | 训练任务编号 |
| `model_name` | 模型名称 |
| `dataset_id` | 训练数据集编号 |
| `status` | 任务状态 |
| `params_json` | 参数摘要 |
| `created_at` | 创建时间 |
| `finished_at` | 完成时间 |
| `version_id` | 对应版本编号 |
| `artifact_dir` | 产物目录 |

## 6. 模型版本数据设计

模型版本对象建议包括以下字段：

| 字段 | 含义 |
| --- | --- |
| `version_id` | 版本编号 |
| `model_name` | 模型名称 |
| `source_task_id` | 来源任务编号 |
| `metric_summary` | 指标摘要 |
| `param_summary` | 参数摘要 |
| `created_at` | 生成时间 |
| `version_dir` | 版本目录 |
| `description` | 版本说明 |

## 7. 目录设计

### 7.1 系统工作目录

系统工作目录建议划分为：

1. 上传数据包目录。
2. 推理输出目录。
3. 训练输入包目录。
4. 训练版本目录。
5. 日志目录。

### 7.2 推理输出目录

应至少保存：

1. 任务说明文件。
2. 结果文件。
3. 日志文件。
4. 可视化文件。

### 7.3 训练版本目录

应至少保存：

1. 模型文件。
2. 指标文件。
3. 参数说明文件。
4. 训练日志。
5. 版本摘要文件。

## 8. 数据库设计建议

若采用轻量数据库记录任务与版本元信息，可至少建立以下逻辑表：

1. `datasets`
2. `inference_runs`
3. `training_runs`
4. `model_versions`

## 9. 数据约束

1. 任务编号必须唯一。
2. 版本编号必须唯一。
3. 数据集编号与文件目录需要保持一致映射关系。
4. 每个版本应能追溯到来源训练任务。

## 10. 已实现与设计预留

### 10.1 当前已实现

1. 统一数据组织思路。
2. 推理任务与训练任务的记录框架。
3. 推理输出目录和训练版本目录的基本组织方式。

### 10.2 设计预留

1. 更完整的数据血缘追踪信息。
2. 更细粒度的训练过程统计数据。
3. 更复杂的数据版本控制机制。

## 11. 论文表述建议

可直接表述为：

“数据设计的关键在于以统一数据包屏蔽四模型输入差异，并以任务记录和版本目录建立从数据输入到结果输出的完整追踪链路。”
