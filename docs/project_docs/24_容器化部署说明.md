# 容器化部署说明

## 1. 部署范围

当前系统的容器化部署采用“两套编排文件 + 一个独立训练服务”的结构：

1. `docker-compose.web.yml`
   负责前端和后端
2. `web_system_runtime/docker-compose.yml`
   负责推理网关和四个模型推理服务
3. WSL 训练服务
   当前不在 Compose 中，由脚本单独启动

## 2. Web 层容器编排

`docker-compose.web.yml` 包含以下服务：

| 服务名 | 作用 | 端口 |
| --- | --- | --- |
| `web-backend` | FastAPI 后端 | `9000:9000` |
| `web-frontend` | Vue 前端静态发布容器 | `5173:80` |

### 2.1 `web-backend` 关键配置

后端容器主要配置如下：

1. `WEB_SYSTEM_DATABASE_URL=sqlite:////app/data/web_system.db`
2. `WEB_SYSTEM_RUNTIME_ROOT=/workspace/web_system_runtime`
3. `WEB_SYSTEM_WORKSPACE_ROOT=/workspace/repo`
4. `WEB_SYSTEM_GATEWAY_URL=http://host.docker.internal:8000`
5. `WEB_SYSTEM_TRAINING_URL=http://host.docker.internal:8011`
6. `WEB_SYSTEM_GATEWAY_RUNTIME_ROOT=/workspace/web_system_runtime`

### 2.2 `web-backend` 关键挂载

| 宿主机路径 | 容器路径 | 用途 |
| --- | --- | --- |
| `./web_system_backend/data` | `/app/data` | SQLite 数据库 |
| `./web_system_backend/outputs` | `/app/outputs` | 下载输出 |
| `./web_system_runtime` | `/workspace/web_system_runtime` | 运行时目录 |
| `./benchmark_factory` | `/workspace/repo/benchmark_factory` | 模型依赖 |
| `./DualSyn` | `/workspace/repo/DualSyn` | 模型仓库 |
| `./MFSynDCP` | `/workspace/repo/MFSynDCP` | 模型仓库 |
| `./MVCASyn` | `/workspace/repo/MVCASyn` | 模型仓库 |
| `./MTLSynergy` | `/workspace/repo/MTLSynergy` | 模型仓库 |

### 2.3 `web-frontend` 说明

前端容器将构建后的静态资源发布到容器内 Web 服务器，对外暴露 `5173` 端口，作为浏览器访问入口。

## 3. 推理运行时容器编排

`web_system_runtime/docker-compose.yml` 包含以下服务：

| 服务名 | 作用 | 端口 |
| --- | --- | --- |
| `gateway` | 推理网关 | `8000:8000` |
| `dualsyn-service` | DualSyn 推理服务 | 容器内部 |
| `mfsyndcp-service` | MFSynDCP 推理服务 | 容器内部 |
| `mvcasyn-service` | MVCASyn 推理服务 | 容器内部 |
| `mtlsynergy-service` | MTLSynergy 推理服务 | 容器内部 |

### 3.1 网关职责

网关负责：

1. 接收后端发来的推理请求。
2. 转发到各个模型服务。
3. 汇总任务状态与输出位置。

### 3.2 运行时挂载

推理运行时编排会挂载：

1. `web_system_runtime/`
2. `benchmark_factory/`
3. `DualSyn/`
4. `MFSynDCP/`
5. `MVCASyn/`
6. `MTLSynergy/`

## 4. 训练服务说明

当前训练服务没有纳入容器化编排文件，而是通过以下命令启动：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system.ps1 -StartTrainingService
```

该服务在 WSL 中运行，并默认通过：

```text
http://127.0.0.1:8011
```

对外提供训练任务和模型版本接口。

## 5. 启动命令

### 5.1 启动 Web 层容器

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system_docker.ps1
```

已构建镜像时可使用：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system_docker.ps1 -NoBuild
```

### 5.2 启动推理运行时

```powershell
docker compose -f .\web_system_runtime\docker-compose.yml up -d --build
```

### 5.3 启动训练服务

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_web_system.ps1 -StartTrainingService
```

## 6. 访问地址

| 对象 | 地址 |
| --- | --- |
| 前端 | `http://127.0.0.1:5173` |
| 后端 | `http://127.0.0.1:9000/health` |
| 系统概览 | `http://127.0.0.1:9000/api/system/summary` |
| 推理网关 | `http://127.0.0.1:8000/health` |
| 训练服务 | `http://127.0.0.1:8011/health` |

## 7. 部署说明

### 7.1 为什么拆成两套 Compose

当前拆分的原因如下：

1. Web 层与运行时层生命周期不同。
2. 前后端调试频率高，推理镜像构建成本更高。
3. 训练服务当前依赖 WSL 环境，不适合直接并入现有 Compose。

### 7.2 后端如何连接运行时

后端在容器内通过：

1. `host.docker.internal:8000` 访问推理网关
2. `host.docker.internal:8011` 访问训练服务
3. `/workspace/web_system_runtime` 读取共享运行时目录

同时使用 `extra_hosts` 将 `host.docker.internal` 映射为宿主机网关，以保证容器内后端能够访问宿主机上的运行时服务。

## 8. 部署结论

当前原生 Web 系统的容器化部署结论如下：

1. Web 层由 `docker-compose.web.yml` 管理。
2. 推理运行时由 `web_system_runtime/docker-compose.yml` 管理。
3. 训练服务单独在 WSL 运行。
