# 模块设计说明

## 1. 模块划分原则

系统模块划分遵循“职责单一、输入清晰、输出可追溯”的原则。整体上将系统拆分为页面模块、接口模块、任务模块、数据模块、模型执行模块和版本模块。

## 2. 页面模块

### 2.1 系统总览模块

职责：

1. 展示系统核心统计信息。
2. 展示服务状态与最近任务。
3. 为答辩演示提供整体入口。

输入：

1. 统计接口返回值。
2. 最近任务列表。
3. 服务状态摘要。

输出：

1. 概览卡片。
2. 状态面板。
3. 最近任务面板。

### 2.2 数据集管理模块

职责：

1. 上传标准数据包。
2. 查看数据集清单。
3. 展示数据集元信息。

### 2.3 推理任务模块

职责：

1. 创建推理任务。
2. 查询任务状态。
3. 查看推理输出和结果摘要。

### 2.4 训练任务模块

职责：

1. 创建训练任务。
2. 查看训练任务详情。
3. 关联训练版本信息。

### 2.5 版本管理模块

职责：

1. 浏览训练版本目录。
2. 查看版本指标、参数和文件清单。
3. 提供模型成果归档视角。

## 3. 接口模块

### 3.1 系统总览接口模块

负责汇总数据集数量、任务数量、版本数量和服务状态。

### 3.2 数据集接口模块

负责处理上传、列表、详情查询和校验结果返回。

### 3.3 推理接口模块

负责推理任务创建、列表查询、详情查询和取消操作。

### 3.4 训练接口模块

负责训练任务创建、列表查询、详情查询和取消操作。

### 3.5 版本接口模块

负责版本列表查询、版本详情查询和文件信息展示。

## 4. 任务编排模块

### 4.1 推理任务编排

处理步骤：

1. 接收前端提交的模型与数据参数。
2. 校验数据集与参数完整性。
3. 生成任务记录。
4. 调用推理服务组件。
5. 回写状态与输出位置。

### 4.2 训练任务编排

处理步骤：

1. 接收训练参数与数据输入。
2. 生成训练任务记录。
3. 调用训练服务组件。
4. 记录版本目录信息。
5. 返回任务与版本关联关系。

## 5. 数据组织模块

### 5.1 统一数据包模块

职责：

1. 规范四模型共享输入字段。
2. 描述药物、细胞系、样本和标签信息。
3. 为推理与训练提供统一入口。

### 5.2 系统工作目录模块

职责：

1. 保存上传数据包。
2. 保存推理输出。
3. 保存训练输入包。
4. 保存训练版本。
5. 保存运行日志与说明文件。

## 6. 模型执行模块

### 6.1 推理服务组件

职责：

1. 接收标准化推理请求。
2. 将请求转换为模型执行参数。
3. 驱动模型执行组件完成推理。
4. 回传结果与状态。

### 6.2 训练服务组件

职责：

1. 接收标准化训练请求。
2. 驱动模型执行组件完成训练。
3. 收集模型文件、指标文件和日志。
4. 生成训练版本目录。

### 6.3 模型执行组件

职责：

1. 面向 `DualSyn`、`MFSynDCP`、`MVCASyn`、`MTLSynergy` 提供统一接入点。
2. 处理模型特定的输入差异。
3. 输出统一格式的结果与版本产物。

## 7. 版本管理模块

职责：

1. 按模型和时间归档训练产物。
2. 保存模型文件、指标文件、说明文件和日志。
3. 支持从版本追溯到训练任务。

## 8. 模块间关系

1. 页面模块通过接口模块访问业务能力。
2. 接口模块通过任务编排模块驱动推理与训练。
3. 任务编排模块依赖统一数据包模块和模型执行模块。
4. 版本管理模块承接训练结果归档。

## 9. 当前实现与后续扩展

### 9.1 当前实现

1. 页面访问链路完整。
2. 接口分类清晰。
3. 推理任务管理机制可用。
4. 训练任务与版本展示机制已形成。

### 9.2 后续扩展方向

1. 增加更细粒度的任务监控。
2. 丰富结果可视化方式。
3. 增强训练过程中的资源监测与异常恢复能力。

## 10. 论文表述建议

本章内容适合在论文中概括为：

“模块设计的重点不在于增加页面数量，而在于围绕统一数据包、统一任务编排和统一版本目录建立稳定的软件边界，使四个药物协同预测模型能够在同一系统框架下协同运行。”
