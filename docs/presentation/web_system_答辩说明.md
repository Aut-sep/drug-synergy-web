# Vue3 版系统答辩说明

## 1. 为什么要从 Streamlit 升级

原系统已经有完整的模型推理、训练、日志和打包能力，但界面层主要依赖 Streamlit。  
升级后的目标不是重写模型，而是把它改造成更标准的前后端分离系统。

## 2. 新系统架构

系统分为四层：

1. 前端展示层  
   使用 `Vue3 + TypeScript + Element Plus` 实现页面交互。

2. 前端状态管理层  
   使用 `Pinia` 管理跨页面共享状态，例如数据集列表、推理任务状态、训练任务状态、模型版本列表。

3. 后端业务层  
   使用 `FastAPI` 提供统一 API，并维护自己的数据库表，记录数据集和任务。

4. 旧模型服务层  
   继续复用 `streamlit_system` 里已有的推理网关和训练服务，保证模型逻辑不需要重新实现。

## 3. Pinia 在这个系统里做什么

`Pinia` 不是页面库，而是前端的状态仓库。

在本系统中：

- `datasets` store：保存数据集列表和当前选中的数据集
- `inference` store：保存推理任务列表和推理任务刷新逻辑
- `training` store：保存训练任务列表和模型版本列表
- `app` store：保存首页系统摘要

答辩时可以说：

`Pinia` 用于集中管理多个页面共享的数据，避免组件之间层层传值，使前端状态更清晰。

## 4. FastAPI 在这个系统里做什么

新的 `FastAPI` 后端不是简单转发页面请求，而是承担了系统层职责：

- 建立自己的数据库
- 记录数据集信息
- 记录推理任务和训练任务
- 对接旧推理网关
- 对接旧训练服务
- 给前端返回统一格式的数据

答辩时可以说：

`FastAPI` 作为系统业务层，对前端提供 RESTful API，同时桥接已有模型服务，实现系统管理和任务编排。

## 5. 一个推理任务是怎么跑通的

1. 用户在 `InferenceView` 选择数据集、模型和模型版本
2. Vue3 页面调用后端 `/api/inference-runs`
3. FastAPI 后端查询数据库里的数据集路径
4. FastAPI 后端请求旧推理网关 `/runs`
5. 旧推理网关调用已有模型推理逻辑
6. 新后端把远端任务 ID、状态和日志摘要写入自己的数据库
7. 前端通过轮询刷新任务状态

## 6. 一个训练任务是怎么跑通的

1. 用户在 `TrainingView` 选择训练数据集和训练参数
2. Vue3 页面调用后端 `/api/training-runs`
3. FastAPI 后端向旧训练服务发起训练请求
4. 旧训练服务在原有环境中运行训练脚本
5. 训练状态定期被新后端同步到本地任务表
6. 训练完成后，新版本可在 `VersionsView` 中查看

## 7. 这一版相比 Streamlit 增加了哪些“真正系统”的内容

- 前后端分离
- 页面路由
- 统一 API
- 数据库持久化
- 任务中心
- 状态管理
- 可扩展的数据集管理
- 对旧模型服务的桥接封装

## 8. 当前代码目录

- `web_system_frontend/`：Vue3 前端
- `web_system_backend/`：FastAPI 后端
- `streamlit_system/`：旧系统与模型服务实现
